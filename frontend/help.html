Supported subtitling specifications

Focus of Subcheck is at the moment IMSC1 (TTML Profiles for Internet Media Subtitles and Captions 1.0)[1]. IMSC1 is normatively referenced amongst others by the SMPTE Interoperable Master Format (IMF)[2], ATSC 3.0 [3] and the MPEG Common Media Application Format standard (CMAF)[4]. IMSC1 is also a superset of EBU-TT-D which is normatively referenced amongst others by HbbTV 2.0.1 [5], DVB DASH [6], the BBC Guidelines [7] and DVB TTML in TS delivery [8]. Therefore, IMSC is currently seen as the most promising candidate for a worldwide subtitling and captioning format, especially for online delivery.

Subcheck also includes an example how “house styles” and guidelines could be checked. See below for more information about how Subcheck validates against the public available Netflix guidelines.

In the following, some more information is given about each of the specifications.

TTML

Regardless which TTML profile is used: All documents must conform to the constraints in TTML 1 [9]. Subcheck therefore not only includes checks on the extra constraints added by the IMSC profile but also more general ones from TTML 1. Most of the TTML constraints that are validated by Subcheck are based on the XML Schema published by W3C. In addition, some checks on the significance for attributes are added. In operation, the semantic significance of attributes gets very important. Although it is not a conformance error to define an attribute on an element where it is not significant, in most cases it points to a misunderstanding of the specification and could lead to a presentation result not intended by the document author.

IMSC1 Common

IMSC1 has two profiles: the Text Profile [10] and the Image Profile [11]. Subcheck currently does not include constraints that are specific to the Image Profile.

Some of the IMSC1 constraints apply to both profiles, the Image and the Text Profile. They are summarized in IMSC1 under Common Constraints [12]. Subcheck provides the option to only check the Common Constraints. If an error is reported for the “IMSC1 Common”, this would be an error for the Text as also for the Image Profile.

IMSC1 Text

Subcheck validates most of the conformance rules imposed by the IMSC1 Text Profile [10]. It excludes a few rules, mostly for one of the two following reasons: The related feature of the constraint is rarely used (or not used at all) or there haven’t been found evidence that subtitle files currently used in operation break the constraint.

An example for the first case is the feature #progressivelyDecodable [13]. This feature is planned to be deprecated in future versions of IMSC [14]. An example for the second case are constraints made by the Hypothetical Render Model (HRM)[15].

If the IMSC1 Text Profile is activated, all constraints made by the IMSC1 Common are also automatically included (even if the IMSC Common profile checkbox is not activated).

IMSC 1.0.1 Common

IMSC 1.0.1 [16] is a minor update of IMSC. It adds two new features and includes editorial clarifications (see below IMSC 1.0.1 Text profile). Although the specification has not reached final recommendation status yet, it is published as Candidate Recommendation and implementers starting with IMSC should take IMSC 1.0.1 as a base reference.

Apart from the URL of the specification reference the Common Constraints of IMSC 1.0.1 are the same as the Common Constraints of IMSC 1.0.

IMSC 1.0.1 Text

IMSC 1.0.1 Text [17] adds two new features: #fillLineGap [18] and #activeArea[19]. Subcheck tests significance and the data type of the related attributes.

Apart from constraint checks of these two new features and the URIs of the specification reference the constraints for the Text Profile of IMSC 1.0.1 are the same as for the Text Profile of version 1.0.

Netflix Guideline

The integration of rules defined by the public available Netflix Timed Text Style Guide (Timed Text Style Guide: General Requirements) [20] is used as an example how arbitrary “House Guidelines” could be tested on top of constraints made by technical specifications. Amongst others Subcheck tests if all text/characters are included in the NETFLIX Glyph List [21].

Note that the integration of the Netflix Guideline does not signal any endorsement by NETFLIX nor that files that pass the Subcheck validation would be accepted by NETFLIX.

Usage

To check a subtitle file, the file could be moved by drag-and-drop onto the marked area. Alternatively, a click in the marked area opens a dialog to select a file from a local drive. The analysis starts immediately.

Before and after the analysis the results can be filtered by various criteria. These criteria are grouped by categories such as “Specifications” or “Error Level”. The filter settings can be opened and closed by clicking on the button “Filter Options”.

If more than one criterion of a category (e.g. “IMSC1-Text” and “Netflix General” for the category “Specifications”) is selected, all messages that apply to one of the select criteria will appear in the result (e.g. all messages that apply to “IMSC1-Text” or “Netflix-General”).

If criteria from distinct categories are selected (e.g. “IMSC1-Text” in the category “Specifications” and “Error” in the category “Error Level”), only those messages that match both criteria will appear in the result (e.g. only messages that signal errors in the IMSC1 Text Profile).

Result list

At the beginning basic information regarding the complete subtitle file, e.g. file size or total number of words, is shown.

The message list contains one entry per violated constraint. Error level and a short constraint title are displayed. By clicking on an entry further information is shown in multiple tabs.

The “Description” tab summarizes the assumption or assertion that was met.

The “More Information” tab provides a detailed description (if available) and tags the message (e.g. regarding error type or editorial constraints).

The “Specification” tab shows all specifications that directly or indirectly refer to the constraint that was violated. Each reference indicates the version, the exact location(s) and, if available, the URI of the specification document. In addition the relevant part of the specification text is quoted.

The “Error Details” tab lists all error locations in the checked source document. Each location is shown as an syntax highlighted XML fragment. Below the XML fragment an XPath expression points to the affected XML node and can be used for further processing, e.g. in an external XML editor. If a constraint is violated at more than 50 locations, only the first 50 locations are shown.
